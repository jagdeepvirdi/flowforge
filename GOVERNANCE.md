# Project Governance

## Current Status

FlowForge is currently maintained by a single maintainer: **Jagdeep Singh Virdi** ([@jagdeepvirdi](https://github.com/jagdeepvirdi)).

## Decision Making

As a solo open-source project, decisions about direction, features, and breaking changes are made by the maintainer. Community input is welcome via GitHub Issues and Discussions — significant changes will be discussed publicly before implementation.

## Roles

| Role | Who | Responsibilities |
|---|---|---|
| **Maintainer** | @jagdeepvirdi | Reviews PRs, merges changes, cuts releases, manages security disclosures |
| **Contributor** | Anyone | Opens issues, submits PRs, improves documentation |

## Contribution Process

1. Open a GitHub Issue to discuss a proposed change before building it
2. Fork the repository and create a feature branch
3. Submit a Pull Request — the maintainer will review within 7 days
4. All PRs require at least one approval before merge (branch protection enforced)

## Continuity

If the maintainer becomes unavailable for an extended period:

- The repository remains publicly available and forkable under the MIT License
- Any contributor with demonstrated history may request maintainer access via a GitHub Issue
- The MIT License explicitly permits the community to fork and continue the project independently

## Versioning

FlowForge follows [Semantic Versioning](https://semver.org). Breaking changes increment the major version. Release notes are published for every release.

## Code of Conduct

All participants are expected to follow the [Code of Conduct](CODE_OF_CONDUCT.md).
