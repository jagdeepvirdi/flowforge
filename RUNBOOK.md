# FlowForge — Runbook

Operational reference for developers and self-hosters. Covers environment setup, database migrations, starting the system, and common CLI tasks.

---

## 1. Prerequisites

| Requirement | Version | Notes |
|---|---|---|
| Python | 3.11+ | Use a virtual environment |
| PostgreSQL | 14+ | FlowForge config DB |
| Oracle Instant Client | 21+ | Only if using Oracle data sources |

---

## 2. Environment Setup

### 2.1 Install dependencies

```bash
pip install -e .             # Core
pip install -e ".[all]"      # Core + Gmail + Drive + M365 + PDF
pip install -e ".[dev]"      # Adds pytest, ruff
```

### 2.2 Configure `.env`

```bash
cp .env.example .env
```

Minimum required variables:

```env
FLOWFORGE_DB_URL=postgresql://flowforge:yourpassword@localhost:5432/flowforge
FLOWFORGE_SECRET_KEY=<generate: python -c "import secrets; print(secrets.token_hex(32))">
FLOWFORGE_USERNAME=admin
FLOWFORGE_PASSWORD=<bcrypt hash: python -c "import bcrypt; print(bcrypt.hashpw(b'yourpassword', bcrypt.gensalt()).decode())">
```

The `FLOWFORGE_DB_URL` controls which PostgreSQL instance stores all FlowForge config (pipelines, schedules, run history). This is **not** your data source database — those are configured in the UI.

## 3. Database Setup

FlowForge uses **Alembic** for schema migrations. There are two scenarios:

### 3.1 Fresh install (no existing tables)

Create the PostgreSQL database, then apply all migrations:

```bash
createdb flowforge        # or use psql / pgAdmin
flowforge db upgrade      # creates all 11 tables + alembic_version
```

After `db upgrade`, start the server — the admin user is seeded automatically from `FLOWFORGE_USERNAME` / `FLOWFORGE_PASSWORD` in `.env`.

### 3.2 Existing database (tables already created by an older version)

If the database was created by a previous FlowForge version that used `db.create_all()` (before Alembic was added), the tables exist but Alembic has no record of them. Tell Alembic the DB is already at the baseline without re-running any DDL:

```bash
flowforge db stamp head
```

Verify it worked:

```bash
flowforge db current
# Expected output: 0001 (head)
```

From this point forward, use `flowforge db upgrade` for all future schema changes.

### 3.3 Applying new migrations (day-to-day)

When a new version of FlowForge adds a schema change:

```bash
git pull
flowforge db upgrade      # apply any new migration scripts
flowforge web             # restart the server
```

### 3.4 Creating a migration (for developers)

After editing `flowforge/db/models.py`, generate a migration:

```bash
flowforge db revision -m "add retry_count to pipeline_steps" --autogenerate
# Review the generated file in flowforge/db/migrations/versions/
flowforge db upgrade
```

Always review autogenerated migrations before applying — Alembic detects most changes but can miss some (e.g., check constraint changes).

## 4. Starting FlowForge

### 4.1 Web server only

```bash
flowforge web
# Starts on http://localhost:5000 (override with FLOWFORGE_PORT)
```

### 4.2 Scheduler only

```bash
flowforge schedule
# Runs the APScheduler daemon — picks up pipelines with a cron schedule
```

### 4.3 Both together (development)

Run in two separate terminals:

```bash
# Terminal 1
flowforge web

# Terminal 2
flowforge schedule
```

For production, use a process manager (systemd, supervisor, Docker Compose). See the example `docker-compose.yml` in the repo root.

### 4.4 Full startup sequence (clean machine)

```bash
# 1. Activate virtualenv
source .venv/bin/activate       # Windows: .venv\Scripts\activate

# 2. Confirm env is loaded
cat .env | grep FLOWFORGE_DB_URL

# 3. Apply any pending migrations
flowforge db upgrade

# 4. Check the current revision
flowforge db current             # should show: <revision> (head)

# 5. Start the web server
flowforge web
```

## 5. CLI Reference

| Command | What it does |
|---|---|
| `flowforge web` | Start the web server |
| `flowforge schedule` | Start the scheduler daemon |
| `flowforge run "Pipeline Name"` | Run a pipeline immediately from the CLI |
| `flowforge list` | List all pipelines and their last status |
| `flowforge validate "Pipeline Name"` | Validate a pipeline's step config without running |
| `flowforge export "Pipeline Name"` | Export pipeline to YAML (stdout or `-o file.yaml`) |
| `flowforge import pipeline.yaml` | Import pipeline from YAML; use `--overwrite` to replace existing |
| `flowforge cleanup` | Delete generated output files older than TTL |
| `flowforge db upgrade [rev]` | Apply pending migrations (default: head) |
| `flowforge db downgrade <rev>` | Revert to a previous revision (e.g. `-1`) |
| `flowforge db stamp <rev>` | Mark DB as being at a revision without running DDL |
| `flowforge db current` | Show the currently applied revision |
| `flowforge db revision -m "msg"` | Generate a new migration script |
| `flowforge db revision -m "msg" --autogenerate` | Auto-detect model changes and generate migration |
| `flowforge setup gmail` | Print Gmail OAuth2 setup instructions |
| `flowforge setup microsoft365` | Print Microsoft 365 setup instructions |

## 6. Running the Test Suite

The tests use a **separate** test database — never the production one.

### 6.1 Test database requirements

The test database must exist before running tests:

```bash
createdb flowforge_test
# Grant access to the test user if needed
```

Default test DB URL (set in `tests/conftest.py`):

```
postgresql://flowforge:harpal123@localhost:5434/flowforge_test
```

Override by setting `FLOWFORGE_DB_URL` before running pytest — `conftest.py` picks it up via `os.environ.setdefault`.

### 6.2 Run tests

```bash
pip install -e ".[dev]"
pytest
```

The `conftest.py` session fixture (`apply_migrations`) automatically:
1. Drops all FlowForge tables in the test DB
2. Runs `alembic upgrade head` to rebuild from migrations
3. This gives every test session a clean, migration-verified schema

### 6.3 Run a specific test file

```bash
pytest tests/test_pipelines.py -v
```

## 7. Import / Export Pipelines

### Export

```bash
# Print YAML to stdout
flowforge export "Monthly Revenue Report"

# Save to file
flowforge export "Monthly Revenue Report" -o monthly_revenue.yaml
```

Secrets are exported as `***` — they are never written to the YAML file.

### Import

```bash
# Import (fails if pipeline with same name exists)
flowforge import monthly_revenue.yaml

# Overwrite existing
flowforge import monthly_revenue.yaml --overwrite
```

After import, set any secret variables (`***`) manually via the Pipeline Builder UI.

## 8. Notes

- **`.env` is auto-loaded** by the Alembic `env.py` from the project root, so `flowforge db` commands work without manually exporting env vars.
- **Schema changes without Alembic** (e.g. direct `ALTER TABLE`) will cause Alembic's autogenerate to detect drift. Run `flowforge db revision -m "description" --autogenerate` to capture them.
- **M365 tokens** are refreshed automatically before each send — MSAL handles the 1-hour expiry transparently via its in-memory cache.
- **Output file cleanup**: `flowforge cleanup --days 7` deletes generated reports older than 7 days. Can be added as a cron job or scheduled pipeline.
